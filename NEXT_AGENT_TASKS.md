# Технічне завдання (PRD) та Архітектурний Контекст для Агента v2.0

## 1. АРХІТЕКТУРНИЙ КОНТЕКСТ (КРИТИЧНО - НЕ ЛАМАТИ!)
Наступний агент, який буде працювати з кодом, зобов'язаний зберегти наступну логіку:
* **SMTP Відправка:** Здійснюється через шлюз Brevo. Усі секрети винесені у файл `.env` (`SMTP_USER` містить системний логін Brevo, а `SMTP_FROM_ADDR=i0323581@gmail.com` строго у нижньому регістрі для уникнення блокування анти-спуфінг фільтрами).
* **IMAP Перехоплення відповідей:** Щоб користувач не світив свою пошту, бот прикріплює до вихідних листів прихований заголовок `Reply-To: publicinquiry69@gmail.com`. Відповіді чиновників падають на цей ящик, де фоновий `IMAP Watcher` (у `watcher.go`) їх зчитує, знаходить збіг у `sentlog.db` (або json) і надсилає в Telegram.
* **PDF-генерація:** Бот успішно генерує і надсилає PDF-файли через скрипт `generate_foi_pdf.py`. Вкладення при отриманні також успішно парсяться і пересилаються в Telegram.
* **Демонізація:** Бот працює як systemd-сервіс (`info-bot-go.service`). Для застосування змін потрібно компілювати бінарник (`go build`) та перезапускати процес (`pkill -9 info-bot`), після чого systemd підніме його автоматично.

## 2. АНАЛІЗ СЛАБКИХ МІСЦЬ ПОТОЧНОГО ПЛАНУ ТА ЇХ ВИРІШЕННЯ
Під час аудиту першої версії ТЗ було виявлено кілька архітектурних ризиків, які тепер враховані в Завданнях:
1. **Ліміт кнопок у Telegram:** Якщо ми додамо 1000+ розпорядників, Telegram не дозволить вивести їх усі кнопками (ліміт 100 кнопок на повідомлення). Потрібна пагінація або пошук.
2. **Ліміти Brevo:** У нас квота 300 листів на день. Потрібен механізм захисту від спаму та черга (Queue), щоб юзери не вичерпали квоту за годину.
3. **Оптимізація БД:** Зчитування сотень JSON файлів сесій для глобальної статистики "покладе" сервер. Потрібно зробити окремий файл `global_stats.json` або повноцінну SQLite таблицю.

## 3. ЗАВДАННЯ НА НАСТУПНИЙ СПРИНТ

### Завдання 1: Оптимізований Модуль Статистики
* **Сховище:** Створити окрему легку структуру (наприклад `global_stats.json`), куди записуватимуться інкременти (+1 запит, +1 новий юзер) при кожній дії, щоб не перебирати всі папки юзерів при запиті статистики.
* **Для звичайного юзера:** Показувати лише його особисті успіхи (скільки запитів відправив, скільки відповідей отримав).
* **Для Адміністратора (Шефа):** Якщо `UserID == ADMIN_ID`, показувати глобальний дашборд:
   * Загальна кількість унікальних користувачів бота.
   * Загальна кількість успішно відправлених запитів.
   * Активність по модулях.
   * Відсоток використання денного ліміту Brevo (наприклад: Відправлено 45/300).
   * Дані суто аналітичні, без виводу персональної інформації користувачів.

### Завдання 2: Ребрендинг Mini-App та UI
* Агент повинен придумати та запропонувати 5-7 потужних, зрозумілих та маркетингових назв (наприклад: "Прозоро-Бот", "SmartZapyt", "Влада на долоні", "ДержІнфо Hub" тощо).
* Після погодження з Шефом — глобально перейменувати додаток у веб-інтерфейсі (HTML/React), кнопках Telegram та текстах привітань.
* Оновити UX/UI дизайн міні-додатка, щоб він виглядав преміально і сучасно (TailwindCSS, красиві анімації).

### Завдання 3: Масштабування бази розпорядників (Парсер + Пошук)
* **Парсер:** Створити скрипт (Python/Go) для парсингу легальних баз (наприклад, сайту "Доступ до правди" або ЄДР) для автоматичного збору правильних email-адрес держорганів.
* **Модуль імпорту:** Створити скрипт або адмін-команду для злиття спарсених даних з поточною базою `directory.json`.
* **Пошук (КРИТИЧНО):** Оскільки адрес буде >1000, повністю переробити механізм вибору органу в Telegram. Замість кнопок зробити Inline-пошук (`@botname київська рада`) або текстове введення з підказками, щоб не впиратися в ліміти Telegram API.

### Завдання 4: Захист від спаму та Ліміти
* Впровадити Rate Limiter (обмеження: не більше 3 запитів від 1 юзера на годину).
* Додати глобальний лічильник денних запитів (максимум 280 на день), щоб не перевищити жорсткий ліміт Brevo (300/день) і залишати запас для системних тестів.

### Завдання 5: Генеральний аудит та вилов помилок
* Пройтися по всьому коду (code review), перевірити обробку помилок.
* Перевірити поточні 69 адрес на предмет помилок синтаксису (опечатки).
* Відшліфувати тексти повідомлень.
